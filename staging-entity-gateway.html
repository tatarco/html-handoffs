1 — The problem

Before (broken on staging)

  • User logs in via Supabase (impersonation / invite works).
  • SPA still calls Base44: base44.entities.*, /api/apps/public, base44.auth.me().
  • Those endpoints do not exist on *.pages.dev → blank screen or StagingShell placeholder.

After (target)

  • Same React screens and @/api/entities imports.
  • On staging, entities go through entityGateway edge function.
  • Business logic stays in named functions (createCandidate, etc.).
  • Real app shell + /Recruitment loads tenant data under RLS.
Important: Backend staging (DB, migrations, seed, RLS, admin) was largely ready. The gap was frontend data-access wiring, not missing tables.
2 — What “done” means (CONTEXT.md — MVP slice B)
Staging slice B succeeds when: authenticated tenant/session context works, the UI shell loads on production-class infra, RLS rejects cross-tenant access, and at least one real ERP stripe does list + create/update end-to-end — not auth-only scaffolding.

First stripe: Recruitment (/Recruitment) — existing BizFlow subsystem, not a throwaway demo.

4 — Two API channels (when to use which)

Channel A — entityGateway

Generic CRUD with the same shape as Base44 entities.

Opslist, get, filter, create, update, delete
Use forLoading lists, patching fields, simple creates
ExamplesCandidate.list(), Branch.filter({ isActive: true })

Channel B — Named functions

Domain logic, webhooks, notifications, complex validation.

InvokeinvokeBizflow('createCandidate', payload)
Use forAnything that was a Base44 Deno function
ExamplescreateCandidate, resolvePermissions, getAllUsers

Base44 functions are transformed to Supabase edge functions in CI (scripts/transform-functions.mjs).

6 — Security layers (defense in depth)

1. JWT validation

getUserFromJwt — reject unauthenticated requests.

2. User-scoped Supabase client

Anon key + forwarded Bearer. Not service role for gateway traffic.

3. RLS (real tenant boundary)

Every row has company_id. Policies check auth.jwt()->'app_metadata'->>'company_id'. Set server-side on impersonation — users cannot forge it.

4. Entity allowlist

Gateway only accepts tables in configured slice bundles. Blocks arbitrary table names even if RLS were misconfigured.

5. RBAC on gateway (phase 2 — planned)

Map entity + op → permission keys; consult sessionRbacContext before execute (same keys as resolvePermissions).

7 — Allowlist bundles

Configured in supabase/native-functions/entityGateway/allowlist.ts. Override slices via env ENTITY_GATEWAY_SLICES=recruitment,core.

BundleTables (snake_case)MVP

9 — Auth & permissions on staging

AuthProvider branch

When isStagingDataRuntime(), AuthProvider reads the Supabase session (no /api/apps/public, no base44.auth.me()).

Maps JWT user → { id, email, role, company_id } for components that expect useAuth().

AppEntry

After login: wraps children with AuthProvider + PermissionsProvider + NavigationTracker + full AuthenticatedApp routes.

StagingShell removed from the happy path.

Permissions

PermissionsProvider calls invokeBizflow('resolvePermissions') on staging → Supabase resolvePermissions edge function → sessionRbacContext.

Recruitment page checks can('recruitment.cru_candidate'). Admins (isAdmin: true) bypass permission checks in the UI.

14 — Why not browser → Supabase directly?
  • Single choke point for allowlist, auditing, and future RBAC.
  • Hides PostgREST table layout from the client.
  • Matches Base44 “entities behind API” — one import path in entities.js.
  • RLS still does the real work — gateway is policy + ergonomics, not a replacement for tenant isolation in the database.
15 — How to verify after deploy
  1. Admin → impersonate tenant user → lands on bizflow-erp-staging.pages.dev
  2. Open /Recruitment — candidate list loads (empty or seeded data)
  3. Add or edit a candidate — create via createCandidate, patch via gateway
  4. Second tenant (Rival Co) — same flow, must not see Humongous rows (RLS E2E)

Probe gateway directly:

curl -X POST "$STAGING_SUPABASE_URL/functions/v1/entityGateway" \ -H "Authorization: Bearer $ACCESS_TOKEN" \ -H "Content-Type: application/json" \ -d '{"entity":"Candidate","op":"list","limit":5}'
Data model reminder (JSONB)
CREATE TABLE candidate ( id UUID PRIMARY KEY, company_id UUID NOT NULL, doc JSONB NOT NULL, -- all fields created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ );

168 Base44 entities → 168 Postgres tables. @bizflow/client unwraps { id, doc } into flat objects { id, ...fields } so React code unchanged.

update() uses jsonb_patch RPC — merge patch, never wipes unmentioned fields.
