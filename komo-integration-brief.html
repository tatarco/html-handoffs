Humondial × Komo — Integration Brief

How and when Humondial sends data to Komo. For validation by Komo team.

Live as of May 28, 2026

Tier Mapping

Humondial has 5 loyalty tiers. Each maps to a Komo group by name. These are the exact strings sent in every API call.

# Humondial Tier Name Points Required Komo Group Value Sent
1 Bench Player
Entry level		 0 pts (all new members) Tier1
2 Starter 250 pts Tier2
3 Striker 500 pts Tier3
4 Top Scorer 750 pts Tier4
5 Humondial Champion 1,050 pts Tier5

API Calls — When & What

There are two types of calls Humondial makes to Komo. All calls use the same API key and go to api.prod.bcomo.com.

📲
New Member Registration ONCE PER USER
Triggered when a player completes OTP login for the first time

When it happens: A user enters their phone number on the Humondial loyalty app and verifies their OTP. This is their very first login — they don't exist in Komo yet.

What Humondial sends to Komo: A registration call to create the member, immediately followed by a group association to place them in Tier1 (the entry tier).

Endpoint 1 — Create member:

POST /api/v4/advanced/registration/quick
{
  "customer": {
    "firstName": "חבר",
    "lastName": "הומונדיאל",
    "phoneNumber": "05XXXXXXXX"
  }
}

Endpoint 2 — Assign to Tier1 (immediately after):

POST /api/v4/advanced/submitEvent
{
  "customers": [{ "phoneNumber": "05XXXXXXXX" }],
  "event": {
    "type": "group_association",
    "data": { "StringValue1": "Tier1" }
  }
}

Headers on both calls:

Headers
x-api-key: [KOMO_API_KEY]
x-source-name: tier1        ← registration call only
x-Source-Type: Humondial    ← group_association call only
x-Source-Name: Side3        ← group_association call only
🔄
Existing Komo Member — First Humondial Login ONCE PER USER
Triggered when the phone already exists in Komo at the moment of first Humondial login

When it happens: The registration call returns a "phone already exists" response (error code 4001014). This means the user is already in Komo from another program.

What Humondial does: Skips re-registration, awards the user a welcome bonus of points (250 pts — the Tier2 threshold) on the Humondial side, and sends a group association to move them directly to Tier2 in Komo.

POST /api/v4/advanced/submitEvent — move to Tier2
{
  "customers": [{ "phoneNumber": "05XXXXXXXX" }],
  "event": {
    "type": "group_association",
    "data": { "StringValue1": "Tier2" }
  }
}
🏆
Tier Upgrade ONGOING
Triggered every time a player crosses a tier threshold through activity

When it happens: A player earns enough points through venue visits, predictions, deliveries, or table bookings to cross into the next tier. The upgrade is detected after each of these four actions.

Activities that can trigger a tier upgrade:

  • Grading a match prediction result
  • Redeeming a venue check-in code at the branch
  • Submitting a match prediction
  • Recording a table booking

What is sent: A single group association event with the player's new tier value.

POST /api/v4/advanced/submitEvent — tier upgrade example (Tier1 → Tier2)
{
  "customers": [{ "phoneNumber": "05XXXXXXXX" }],
  "event": {
    "type": "group_association",
    "data": { "StringValue1": "Tier2" }
  }
}

The value in StringValue1 will be Tier2, Tier3, Tier4, or Tier5 depending on which tier the player just unlocked.

⚠️
Action needed from Komo: Please confirm that the group names Tier1, Tier2, Tier3, Tier4, and Tier5 exist and are correctly configured on your side (via group_association events). If your system uses different names, let us know and we will update our config immediately — no code changes required.
