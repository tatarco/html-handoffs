Overview

The migration runs on your machine, not as a Base44 edge function. It is deliberately slow and checkpointed: one employee at a time, files one by one, with every outcome written to progress.csv before moving on.

┌─────────────┐ ┌──────────────┐ ┌─────────────┐ ┌──────────────────┐ │ SNAPSHOT │ ──► │ ANALYZE │ ──► │ progress.csv│ ──► │ migrate --live │ │ (disk cache)│ │ build plan │ │ + cache.json│ │ employee by emp │ └─────────────┘ └──────────────┘ └─────────────┘ └──────────────────┘ ▲ │ │ Re-run same command resumes from CSV ◄────────────┘ │ --refresh-snapshot only when Monday/BizFlow master data changed

Current plan (stakeholder rules ON): ~1,178 employees to create, 11 skipped (already in BizFlow), 7 blocked (must fix in Monday first), ~15,600 files.

Pre-flight checklist

  1. Resolve all 7 blockers in Monday (or accept they will never import).
  2. Confirm stakeholder mapping rules are final — re-run analyze after any mapping change.
  3. Set env vars in the shell that will run for hours: 
    export BASE44_API_KEY='…'
export MONDAY_API_TOKEN='…'
  4. Ensure getEmployees with migrationBulk: true is deployed (already on main).
  5. Keep the laptop awake; use tmux or screen so a terminal close does not kill the process.
  6. Open the live dashboard in a browser (auto-refreshes every 5s): 
    open .scratch/monday-migration/status.html

Phase 3 — Live migrate (actual import)

node tools/monday-migration/run.mjs migrate --live

What the runner does at startup

  1. Loads analyze-cache.json (must exist — run analyze first).
  2. Loads progress.csv.
  3. Loads employee snapshot from disk (does not re-fetch BizFlow unless you ran analyze with refresh).
  4. Builds queue: all rows where run_status is pending, partial, failed, or in_progress.
  5. Skips blocked rows entirely (plan_status starts with blocked_).

Loop (repeated until queue empty or you stop)

For each queued employee, in CSV order:

  1. Mark row in_progress → write CSV + dashboard immediately.
  2. Execute employee + file steps (see next section).
  3. Set final status → write CSV + dashboard immediately.
  4. Sleep --delay-ms (default 200ms) before next employee.

When the batch ends, the process exits. Re-run the exact same command to continue. There is no background daemon.

Per-employee steps (inside the loop)

START row (pending / failed / partial) │ ├─ plan_status = blocked_* ──────────────────────────► SKIP (never enters live queue) │ ├─ plan_status = flagged_existing_in_biz (no --force-merge) │ ├─ Link existing employee_id (no update) │ └─ Upload files only if employee_id known │ └─► run_status = skipped_existing │ ├─ run_status = partial (employee already created) │ └─ SKIP create/update ──► upload files only │ └─ otherwise (pending / failed) ├─ No existing employee in snapshot → POST Employee (create) ├─ Existing + --force-merge → PUT Employee (update) └─ employee_id saved to CSV │ ▼ FOR EACH intake file (Monday asset): ├─ Already exists as EmploymentDocument (same name)? → skip (count skipped) ├─ Fetch Monday public URL → download → UploadFile → POST EmploymentDocument └─ On file error: log, count failed, continue other files │ ▼ ALL files OK? ──► complete SOME files failed? ──► partial (employee kept, retry files later) Employee step threw? ──► failed (retry employee + files later)

Run statuses — what they mean

StatusMeaningNext run behavior
pendingReady to import; nothing attempted yet.Create employee + upload all files.
in_progressInterrupted mid-row (crash, Ctrl+C, laptop sleep).Resumes as partial (if employee_id set) or pending.
completeEmployee created/linked and all files uploaded.Skipped permanently (until you manually reset CSV).
partialEmployee exists; one or more files failed.Files only — does not re-create employee. Retries failed files; skips existing doc names.
failedEmployee create/update threw (network, validation, 5xx).Retries employee step + files from scratch.
skipped_existingAlready in BizFlow; files processed per skip rules.Not queued again.
blockedPlan blocker (missing TZ, position, etc.).Never runs in live mode until Monday is fixed and you re-analyze.

Failure recovery playbook

1. Terminal closed / laptop slept / Ctrl+C

Check progress.csv and status.html. Any in_progress row may be partial-complete. Simply re-run:

node tools/monday-migration/run.mjs migrate --live

2. Single employee failed (red in dashboard)

Read error column in CSV or runner.log. Fix root cause (bad data, rate limit). Re-run full migrate — only failed rows retry.

To test one person after fixing Monday:

node tools/monday-migration/run.mjs analyze --tz 123456789
node tools/monday-migration/run.mjs migrate --live --tz 123456789

3. Partial — employee OK, some files failed

Employee record already in BizFlow. Next run uploads only files that are not already EmploymentDocument names. Check files_failed and error in CSV for asset names. Common causes: Monday URL expired, upload timeout.

4. Rate limit (HTTP 429)

BizFlow client auto-retries with backoff (up to 4 retries). If persistent, increase delays:

node tools/monday-migration/run.mjs migrate --live --delay-ms 1000 --file-delay-ms 300

5. Stale analyze cache (mapping changed after analyze)

node tools/monday-migration/run.mjs analyze
node tools/monday-migration/run.mjs migrate --live

Completed rows keep their status; pending rows get updated payloads from the new plan.

6. Need to re-import an already-complete employee

No built-in "reset one row" CLI. Manually edit progress.csv: set run_status=pending, clear employee_id, zero file counters. For existing BizFlow employees use --force-merge to update fields. Duplicates are otherwise avoided.

7. Catastrophic mistake — start CSV fresh

Delete or rename progress.csv and analyze-cache.json, then re-analyze. Risk: may attempt to create employees that already exist if snapshot is stale. Prefer analyze --refresh-snapshot so existing employees are detected.

Timing expectations

Rough lower bound for ~1,178 employees and ~15,600 files (defaults: 200ms between employees, 100ms between files):

StepPer unitTotal (approx)
Employee create~1–3 s API~20–60 min
File upload~2–5 s eachmany hours (bulk of runtime)
Inter-row delays200ms × 1178~4 min

Plan for an overnight run (or multiple sessions using resume). Use --limit N for controlled batches during business hours.

Do not

  • Do not run analyze --refresh-snapshot mid-migrate unless you understand snapshot drift — usually wait until a batch finishes.
  • Do not delete employees in BizFlow mid-run without resetting CSV rows — creates orphan state.
  • Do not run two migrate processes in parallel on the same CSV.
  • Do not expect Base44 function logs — all observability is local CSV + dashboard + runner.log.
  • Do not use --force-merge unless you intentionally want to overwrite existing BizFlow employee fields.
