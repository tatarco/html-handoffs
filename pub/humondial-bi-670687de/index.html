Humondial Promo — Data Access Guide

Everything a BI developer needs to connect to the Humondial football-prediction promotion data: a dedicated read-only account, the full data model, how the pieces talk to each other, the gotchas that will bite you, and a library of ready-to-run analytical queries. READ-ONLY PROMO SCOPE ONLY

Overview

Humondial is a fan engagement promotion: players register on a mobile web app (PWA), predict football match scorelines, earn points, climb tiers and a leaderboard, and redeem perks at restaurant branches. You are being given a curated, read-only window onto that data for analytics and reporting.

Architecture & models — how the pieces talk

Three systems cooperate. You are reading the database at the bottom; knowing the layers above explains why the data looks the way it does.

  ┌──────────────────────────┐   player actions (register, predict, redeem)
  │  Player PWA  (Vite/React)│   ───────────────────────────────────────────┐
  │  humondial-promo         │                                              │

               ▼                                                           │
  ┌──────────────────────────┐   getPlayerCampaignConfig · getLeaderboard  │
  │  Base44 backend functions│   getPromoReporting · syncLiveFixtureScores  │
  │  (biz-flow-erp dev app)  │   syncBranchDailyStats · promoVerifyOtp ...  │
  └────────────┬─────────────┘                                              │
               │ SQL (service role)                                        │
               ▼                                                           ▼
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  Supabase Postgres  (Bizflow-dev · qnbogoyoqbkewkdbhxuy)                  │
  │  promo_* tables  ←─ YOU read these (read-only, masked)                    │
  └─────────────────────────────────────────────────────────────────────────┘
               ▲
  ┌────────────┴─────────────┐
  │  CMS / backoffice        │  campaign config, rules copy (writes promo_campaigns)
  │  (bizflow-erp)           │
  └──────────────────────────┘
  • PWA never touches the DB directly — it calls Base44 functions, which hold the privileged service credentials and enforce the rules.
  • Base44 functions are the only writers. They award points, grade predictions, sync live scores, and roll up branch stats. Every business rule lives here, not in the DB.
  • Supabase Postgres is the system of record. It is a shared multi-tenant ERP database; the promotion lives in the promo_* table family. Your account can see only those.
  • CMS (bizflow-erp) writes campaign configuration into promo_campaigns.
Implication for BI: the DB is the output of business logic, not the logic itself. Points already reflect the rules; you sum them, you don't recompute them. Some tables (promo_leaderboard_cache, promo_stats_cache) are pre-computed snapshots refreshed on a timer — treat them as caches, not truth.

Match status is time-derived. There is no single status column. The points economy

promo_ledger is the heart of the dataset: one row per points event, append-only. A player's true balance is SUM(points) over their non-pending ledger rows. The reason column classifies every event:

reasonMeaningConfig baseRows (sample)
prediction_participationSubmitted a prediction for a match10 pts2,531
prediction_outcomeGot the result right (W/D/L)15 pts701
prediction_bullseyeExact scoreline correct30 pts235
komo_existing_member_bonusEnrollment bonus (already a member)—554
venue_visitVerified visit to a branch65 pts157
table_bookingBooked a table via promo link15 pts144
deliveryPlaced a delivery order via promo link65 pts62

Other configured rewards you may see appear over time: join (50), adr / draw-stripe specials (20 / 10). The authoritative amount is always the points column on the row — never assume the config base, since it can change.

pending = true marks provisional points (e.g. awarded during a live match before final settlement). Exclude pending rows when reporting confirmed balances. idempotency_key guarantees each event is booked once.

Tiers

Players are bucketed into tiers by total points. Configured tiers (ascending): benchstarterstriker top_scorerchampion. The current tier per player is materialized on promo_leaderboard_cache.

Credentials & scope

You have one identity, bi_readonly, provisioned specifically for BI. It is:

  • Read-only — SELECT only. No INSERT/UPDATE/DELETE/DDL anywhere. Verified.
  • Scoped to the promotion — it can read the promo_* analytics tables and nothing else. Other tenants' ERP tables (orders, employees, payroll, …) return permission denied.
  • PII-masked — phone numbers, OTP/code hashes and session tokens are not exposed. Player-level data comes through curated views (see next section).
  • Self-limiting — a 120s statement timeout protects the shared DB from runaway queries.
Handle as a credential. The password / REST token are sent to you privately. Don't paste them into shared docs, tickets, or this page. Ask for rotation if a laptop is lost or someone leaves. This is a dev database — do not load this data into a public dashboard.

Gotchas — read before you trust a number

1. The decoy campaign row. promo_campaigns has two rows. The real one is bf987a45-9783-4507-9a26-acfd5f145473 (8 branches, 5 tiers). A second decoy row f746d606-… exists. An unfiltered limit 1 can grab the wrong one. Always pin the campaign id.
2. REST 1,000-row cap. PostgREST never returns more than 1,000 rows. Aggregate in SQL or use the Postgres connection for full extracts.
3. Status is time-derived. No status column on matches — derive from kickoff_utc, the 5-min lock offset, and presence of final_* scores.
4. Caches lag. promo_leaderboard_cache and promo_stats_cache are periodic snapshots. For exact, current points, aggregate promo_ledger yourself (it's the source of truth).
5. pending points. Exclude pending = true ledger rows from confirmed balances.
6. Attribution is partial. origin_source is null for the majority of players — treat channel mix as "of those tagged."
7. UTC everywhere. All timestamps are UTC; convert to Asia/Jerusalem for business-day reporting: created_at AT TIME ZONE 'Asia/Jerusalem'.
8. Dev data. This is the Bizflow-dev database — it contains test players and may be reset. Don't treat it as production-grade truth or expose it publicly.

Suggested queries — start here

Copy-paste SQL (Postgres path). Every query pins the campaign. Swap in :c := 'bf987a45-9783-4507-9a26-acfd5f145473' or inline the id.

1 · KPI snapshot — the one-screen overview

with c as (select 'bf987a45-9783-4507-9a26-acfd5f145473'::uuid id)
select
  (select count(*) from promo_players_bi)                                          as players,
  (select count(*) from promo_predictions p, c where p.campaign_id=c.id)         as predictions,
  (select count(distinct player_id) from promo_predictions p, c where p.campaign_id=c.id) as predictors,
  (select coalesce(sum(points),0) from promo_ledger l, c where l.campaign_id=c.id and not pending) as points_awarded,
  (select round(100.0*count(*) filter (where outcome='correct')/nullif(count(*) filter (where outcome is not null),0),1)
     from promo_predictions p, c where p.campaign_id=c.id)                       as pct_correct;

2 · Engagement funnel — registered → predicted → won → redeemed

select
  count(distinct pl.id)                                                  as registered,
  count(distinct pr.player_id)                                           as made_a_prediction,
  count(distinct pr.player_id) filter (where pr.outcome='correct')        as won_at_least_once,
  count(distinct pf.player_id)                                           as redeemed_a_perk
from promo_players_bi pl
left join promo_predictions     pr on pr.player_id=pl.id
left join promo_perk_fulfillments pf on pf.player_id=pl.id;

3 · Points awarded by reason (where the economy spends)

select reason, count(*) as events, sum(points) as total_points,
       round(avg(points),1) as avg_points
from promo_ledger
where campaign_id='bf987a45-9783-4507-9a26-acfd5f145473' and not pending
group by reason order by total_points desc;

4 · Daily registrations & new-player points (local time)

select (created_at at time zone 'Asia/Jerusalem')::date as day,
       count(*) as signups
from promo_players_bi
group by 1 order by 1;

5 · Leaderboard top 20 (recompute from the ledger — exact, not cached)

select p.nickname,
       sum(l.points) as points,
       rank() over (order by sum(l.points) desc) as rnk
from promo_ledger l
join promo_players_bi p on p.id=l.player_id
where l.campaign_id='bf987a45-9783-4507-9a26-acfd5f145473' and not l.pending
group by p.nickname
order by points desc limit 20;

6 · Prediction accuracy by match stage (which rounds are "hard")

select m.stage,
       count(*) filter (where pr.outcome is not null)        as graded,
       round(100.0*count(*) filter (where pr.outcome='correct')
             /nullif(count(*) filter (where pr.outcome is not null),0),1) as pct_correct
from promo_predictions pr
join promo_matches m on m.id=pr.match_id
where pr.campaign_id='bf987a45-9783-4507-9a26-acfd5f145473'
group by m.stage order by pct_correct desc nulls last;

7 · Acquisition channel mix (of tagged players)

select coalesce(origin_source,'(untagged)') as source,
       count(*) as players,
       round(100.0*count(*)/sum(count(*)) over (),1) as pct
from promo_players_bi
group by 1 order by players desc;

8 · Branch performance — acquisition vs redemption

select branch_name,
       sum(new_joins)        as joins,
       sum(redemptions)      as redemptions,
       sum(mains_served)     as mains,
       round(100.0*sum(new_joins)/nullif(sum(mains_served),0),1) as join_per_100_mains
from promo_branch_daily_stats
where campaign_id='bf987a45-9783-4507-9a26-acfd5f145473'
group by branch_name order by joins desc;

9 · Retention — return-day cohort (sessions after signup)

with first_seen as (
  select player_id, min(session_date) as d0 from promo_player_sessions group by player_id)
select (s.session_date - f.d0) as day_offset,
       count(distinct s.player_id) as active_players
from promo_player_sessions s join first_seen f using (player_id)
group by 1 order by 1;

10 · Power users — predictions, hit-rate & points in one row

select p.nickname,
       count(pr.*)                                    as predictions,
       count(*) filter (where pr.outcome='correct')   as correct,
       coalesce(sum(l.points) filter (where not l.pending),0) as points
from promo_players_bi p
left join promo_predictions pr on pr.player_id=p.id
left join promo_ledger      l  on l.player_id=p.id
group by p.nickname
having count(pr.*) > 0
order by points desc limit 50;
