1. The problem

Staging symptom

entityRbacGateway returns 503 BOOT_ERROR — the Edge Function module fails to load before any request logic runs.

Root cause on main

scripts/transform-functions.mjs did two incompatible things:

  1. Removed const base44 = createClientFromRequest(req) and injected sb / db / dbService instead
  2. Partially rewrote static base44.asServiceRole.entities.Employee → dbService.employee

But entityRbacGateway also uses patterns the transform never rewrote:

const base44 = createClientFromRequest(req); // removed → base44 undefined const ctx = await hydrateRbacFromRequest(base44); // passes undefined const db = base44.asServiceRole.entities as Record<...>; const entityApi = db[entity]; // dynamic access — never rewritten

Deployed code references base44 after it was stripped → module load failure → BOOT_ERROR.

GitHub PR description is stale. It still describes the first commit (regex body rewriting). The actual code after commit 2 uses the shim approach documented below.
2. PR strategy
Library adds a Base44-compatible client (createClientFromRequest + buildPascalEntityMap + $in filters).
Transform stops rewriting function bodies and only swaps the SDK import.
Fixes entityRbacGateway boot failure without regex surgery on entry.ts.

Before (main transform)

  • Strip @base44/sdk import
  • Replace createClientFromRequest(req) with inline Supabase setup
  • Regex rewrite every base44.entities.X → snake_case db.x
  • Inject module-level dbService + _entityProxy

After (this PR)

  • Swap import → @bizflow/client
  • Keep const base44 = createClientFromRequest(req) as-is
  • Keep all base44.* calls unchanged
  • Shim implements the Base44 API shape in the library

packages/bizflow-client/src/createClientFromRequest.ts NEW +52 lines

What: Drop-in replacement for createClientFromRequest from @base44/sdk@0.8.23 on Supabase Edge Functions.

Returns

PropertyImplementation
auth.me()getUserFromJwt(userClient, jwt) — returns null on failure (matches callers that check if (!userRecord))
entitiesbuildPascalEntityMap(userClient) — user-scoped, RLS applies
asServiceRole.entitiesbuildPascalEntityMap(serviceClient) — service role, bypasses RLS

Design choices

  • Service client singleton — one createClient(SUPABASE_URL, SERVICE_ROLE_KEY) per isolate
  • User client per request — JWT from Authorization header forwarded to anon client
  • Env required: SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY

Why it exists

Core fix. Makes all existing base44.* code work unchanged:

hydrateRbacFromRequest(base44) base44.asServiceRole.entities.UserRoleAssignment.filter(...) const db = base44.asServiceRole.entities; db[entity]

Does NOT implement

base44.integrations.* (InvokeLLM, UploadFile, SendEmail). Pre-existing staging gap — not introduced here.

Review flags
  • auth.me() swallows all errors → null. Callers can't distinguish invalid token vs missing company_id.
  • No integrations stub — functions using it will runtime-error (not boot-error) on that code path.
packages/bizflow-client/src/entityGateway.ts +11 lines

What: Adds buildPascalEntityMap(sb) — a Proxy over buildEntityClient.

export function buildPascalEntityMap(sb) { return new Proxy(cache, { get(_, entityName) { const table = entityNameToTable(entityName); // Employee → employee return cache[table] ??= buildEntityClient(sb, table); }, }); }

Why

  • Base44 uses PascalCase entity names (PermissionSet, UserRoleAssignment)
  • Supabase tables are snake_case (permission_set, user_role_assignment)
  • Enables db['Candidate'] and db.Candidate without transform rewriting

Used by: createClientFromRequest for both entities and asServiceRole.entities.

packages/bizflow-client/src/entities.ts +56 −2

What: buildEntityClient().filter() now supports Base44 { $in: [...] } operator.

Before

filter: (query) => sb.from(table).contains('doc', query)

Supabase JSONB contains() cannot express $in. This RBAC query would fail after boot:

PermissionSet.filter({ id: { $in: setIds } })

After — three paths

Query shapeStrategy
id: { $in: [...] }Supabase .in('id', ids), then filter other fields in memory
Other fields with $inFetch rows (with contains prefilter if possible), filter in JS via docMatchesQuery
No $inUnchanged contains('doc', query)

Why: Second bug behind BOOT_ERROR. RBAC hydration loads permission sets by ID list in both entityRbacGateway and managePermissionSets.

Review flags
  • Non-id $in loads potentially large result sets then filters in JS. Fine for PermissionSet; risky on large tables.
  • Only $in supported — not $gt, $ne, etc.
