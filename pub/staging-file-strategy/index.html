1 — Executive summary

The problem in one sentence

Entity rows live in Postgres with company_id RLS, but file bytes still sit on Base44 production CDN — and staging uploads would proxy there too if the shim worked.

✓ Done

Entity tables migrated to Supabase. Tenant RLS on company_id. entityGateway routes CRUD. Auth shell + impersonation in delivery/.

⚠ Partial

UploadFile shim exists but calls the wrong endpoint with the wrong protocol. entityRbacGateway Supabase copy is broken. No storage buckets.

✗ Missing

Private tenant-scoped storage. File registry. Signed read URLs. Backfill of base44.app URLs in migrated doc JSON. RBAC on file bytes.

Goal: Mirror single-tenant ergonomics (UploadFile({ file }) → file_url) while upgrading isolation to private storage + short-lived signed URLs + permission checks on every read.
2 — How single-tenant handles files

Upload API (one integration, ~25 UI call sites)

const { file_url } = await base44.integrations.Core.UploadFile({ file });

Used by invoices, receipts, procedures, suppliers, recruitment, branding, feedback, payroll attachments, and more. Returns an opaque CDN URL stored inline on entity fields.

Where URLs are stored (not a separate File entity)

Security model (production)

  • Upload — Base44 user session; SDK sends auth.
  • Metadata — entityRbacGateway + entity ACLs gate who sees the URL (e.g. invoices.view).
  • Read — UI uses URL directly in <img>, <a href>, react-pdf. No second auth hop.

Implication

RBAC protects the pointer, not the bytes. The URL is the capability token. Acceptable-ish in single-tenant; not acceptable in multi-tenant staging where cross-tenant URL leakage is catastrophic.

4 — Active leak vectors

1 — Migrated production URLs

DB rows expose base44.app file URLs. Anyone with the URL can fetch bytes — not tenant-scoped, not RBAC-gated at read time.

2 — Upload proxy to prod

uploadBotFileToStorage forwards to production Base44 UploadFile. New staging uploads would write to prod CDN.

3 — Weak entity gateway

Staging entityGateway has no permission matrix — only company_id RLS. Weaker than production entityRbacGateway.

4 — URL-as-secret model

UI renders raw URLs. No signed-URL expiry, no read-time permission check on file bytes.

Treat staging as containing live leak paths to production blobs until backfill completes and the upload shim is rewired.
5 — Mental model: three separate layers
Files are tenant assets first. Links to business records and branch scope are authorization concerns — not bucket layout. Do not encode entity id, record id, or branch id in the storage path.

Layer 1 — Storage

Only job: tenant isolation. One private bucket. Path is company_id + opaque id (+ optional coarse category folder for ops).

No RBAC logic here. No branch folders. No entity names in paths.

Layer 2 — Registry

file_asset row per blob: who uploaded, mime, size, optional link to parent entity, legacy URL, lifecycle status.

Source of truth for what this file is, not where bytes live on disk (that's derivable from id).

Layer 3 — Authorization

Computed at sign time from: file category defaults, linked parent entity (if any), user's permission set, and parent's branchId only when the parent has one.

Three file lifecycles in the app today

KindExampleUpload RBACRead RBAC
Orphan Invoice drag-drop: upload before AccountingDocument exists Category permission (invoices.create) Uploader + same category perm, until linked

Branch scope — when it applies (not in the path)

Only some parent entities carry branchId. Many file hosts do not:

File hostHas branchId?
AccountingDocument, SupplierAgreementYes — enforce branch scope at sign
Candidate, Procedure, Branding, SystemFeedbackNo — company-wide read per module perm
EmploymentDocument, EmployeeNoteIndirect — resolve via Employee.branchId if needed

Rule: branch is looked up from the linked parent at read time. Never stored as a path segment. Optional denormalized branch_id on file_asset only as a cache after link — not required at upload.

6 — Storage layout (keep it flat)

Recommended — flat + optional category

{company_id}/{file_id}.{ext}

or, if you want bucket browsing without DB:

{company_id}/{category}/{file_id}.{ext}

file_id = UUID v4. category = small enum (see below). Not entity name, record id, or branch.

Avoid — over-fitted paths

{company_id}/{entity}/{record_id}/{field}/{uuid}.ext {company_id}/{branch_id}/...

Couples storage to DB lifecycle, complicates orphan uploads, wrong for company-wide files, breaks when records move or merge.

Category enum (coarse — for ops + default upload perms)

invoices recruitment hr suppliers certification branding feedback bot webhook

Category is a hint passed at upload (shim can infer from call site later). Flat tenant/uuid.ext is valid if you prefer absolute minimum — category then lives only in file_asset.

7 — fileGateway edge function

Single entry point mirroring UploadFile and adding secure reads. Reuses resolveSessionRbacContext from packages/bizflow-client/src/rbac.ts.

OpAuthRBACBehavior
upload Bearer JWT Category permission (e.g. invoices.create) — entity link optional Validate mime + magic bytes + size → write {tenant}/{id}.ext → insert file_asset (orphan ok) → return { file_ref }
link Bearer JWT Entity update permission Attach link_entity/link_id/link_field when UI saves entity field (or inline on create)
sign Bearer JWT Derived: linked parent policy or orphan uploader policy Verify company_id → resolve auth → return signed URL (60–300s TTL)
delete Bearer JWT Matching delete permission Soft-delete registry row; optional storage purge

Upload validation (inherit from uploadBotFileToStorage)

image/jpeg image/png image/webp application/pdf magic-byte verify 15 MB max
8 — Authorization at sign time (not in the bucket)

Decision tree for sign

1. file_asset.company_id === jwt.company_id → else 403 2. If link_entity set: a. Load parent entity b. Check module permission (matrix below) c. If parent.branchId exists AND user scope is branch-limited → parent.branchId ∈ user.branchIds → else 403 3. Else (orphan): a. uploaded_by === user.id OR user has category upload perm 4. Issue signed URL

9 — Delivery shim changes (keep src/ unchanged)

Today — supabaseBase44Shim.js

UploadFile → uploadBotFileToStorage (FormData + Bearer — wrong)

Target

UploadFile → fileGateway { op: 'upload' } return { file_url, file_ref }

Read helper (delivery-only or shared)

resolveFileUrl(urlOrRef) → fileGateway { op: 'sign' } → cached short-lived signed URL

Wire DocumentViewer, FilePreviewModal, and download <a href> links through it. Production Base44 keeps raw URLs unchanged.

10 — File backfill plan
0

Stop the bleeding (day 1)

  • Disable production Base44 proxy in staging uploadBotFileToStorage
  • Audit migrated data: scan doc JSON for base44.app URLs per entity table
  • Treat staging as containing production file URLs until backfill completes
1

Infrastructure (week 1)

  • Create private bizflow-files bucket + file_asset table + RLS
  • Implement fileGateway (upload + sign)
  • Fix UploadFile shim → fileGateway
  • Add resolveFileUrl() and wire viewers
  • Fix entityRbacGateway on Supabase
2

Backfill script (week 1–2)

For each entity table × file field: 1. Scan doc JSON for base44.app / known CDN URLs 2. Download blob (service role / api_key) 3. Re-validate mime + magic bytes 4. Upload to bizflow-files/{company_id}/{category}/{file_id}.ext 5. Insert file_asset (source='migration', legacy_url=original) 6. Rewrite doc field to file_ref or staging URL pattern 7. Log failures → migration_errors (auditable, retryable)

Priority order: AccountingDocument → EmploymentDocument → Candidate.resumeUrl → suppliers → procedures → feedback.

Dual-read period: fileGateway/sign accepts file_ref and legacy base44.app URLs — legacy only after verifying parent entity belongs to requester's tenant.

3

Harden (week 2+)

  • Block new writes of raw external URLs on entity update
  • Expire legacy URL support after backfill completes
  • Add file_access_log audit table
  • E2E: tenant A cannot sign tenant B's file; branch-scoped user blocked on out-of-scope PDF

12 — Recommended next steps
  1. Do not treat file migration as optional — migrated rows are live leak paths.
  2. Implement fileGateway before more staging features touch uploads or preview.
  3. Fix entityRbacGateway on Supabase so file + entity permissions share one RBAC context.
  4. Run URL inventory query across entity tables to size backfill.
  5. Keep src/ unchanged — all file logic in delivery/ + supabase/functions/fileGateway/.
