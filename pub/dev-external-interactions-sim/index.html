Real FillFaster, Meta, the WhatsApp bot, Slack, Tabit and the rest only point at production. On dev & staging those counterparts don’t exist — so we build a believable fake of them that drives the real staging handlers, and we record every in/out interaction in one place.

Two jobs in one tool: (1) a developer test harness to fire inbound events and watch side-effects, and (2) a faithful “fake outside world” anyone can interact with like production — chat the bot in a fake WhatsApp, submit a fake Meta lead, sign a fake document.

Every integration is one of three shapes

PUSH · they call us

Mock = payload injector

Compose a webhook body and POST it to the real staging function. It runs for real, writes entities, fires downstream.

FillFaster signed-doc · Meta/website/public leads · WhatsApp inbound

PULL · we call them

Mock = fake API responder

Point the function’s base-URL env at a mock gateway that returns canned, scriptable responses.

FillFaster PDF download · Meta Graph (lead detail) · Google Places · Tabit (via Supabase)

SINK · we send out

Mock = capture, don’t send

Divert real outbound sends into a recorder so no real WhatsApp/email/Slack leaves staging.

WhatsApp templates · Make webhooks · Meta conversions

The headline: a fake WhatsApp that actually loops

You type in Fake WhatsApp │ POST { phone, textBody, replyId } ▼ ┌─────────────────────────────────────────┐ │ REAL staging fn: handleWhatsAppBotInbound │ ← runs the real state machine, │ ?testMode=true │ reads/writes real entities └─────────────────────────────────────────┘ │ returns BotMessage { text | buttons | list } (synchronous) ▼ Rendered back as the bot’s bubble ───► every hop logged to ExternalInteraction

handleWhatsAppBotInbound?testMode=true already returns the bot reply synchronously — so the loop closes without any real provider, and the conversation exercises the genuine bot logic, sessions and submissions.

  3. Returned BotMessage (text / buttons / list) renders as the bot bubble

Build plan — thin slices, value first

Each phase is independently shippable and demoable on staging.

Phase 0 · Backbone ~½ day

  • New ExternalInteraction entity (direction, integration, event, status, payloadJson, responseJson, relatedEntityType/Id, correlationId, createdAt).
  • Tiny logInteraction() helper (inline-duplicated per function per the entry.ts isolation rule).
  • MOCK_MODE staging env flag.

Phase 1 · Inbound Injector ~1 day

  • Admin route + payload composer with realistic presets for the 4 PUSH flows.
  • Dry-run toggle + test-tenant guard.
  • Immediate testing value for every PUSH integration with zero handler changes.

Phase 2 · Outbound SINK + Mock Gateway ~1–2 days

  • One delivery/edge-functions/mockGateway fn: records calls, returns canned responses.
  • Env-ize the finite list of outbound base-URLs (99digital, Slack, Graph, API-Football, Komo, Places) → point at gateway when MOCK_MODE.
  • No real WhatsApp/email/Slack leaves staging; all captured in the log.

Phase 3 · Fake WhatsApp + Fake World ~2 days headline · end of v1

  • Fake WhatsApp chat loop over handleWhatsAppBotInbound?testMode=true.
  • Fake Meta/website lead form · Fake FillFaster signing page · Fake Slack card.

Phase 4 · (optional, later) minor PULL responders ~½ day

  • Fake Google Places autocomplete + Tabit catalog stub — only if those flows need exercising.
  • Promo / sports (API-Football, Komo, OTP) — excluded, not needed.

Open questions for the morning

  • SINK invasiveness — OK to make the ~6 outbound base-URLs env-configurable in base44/functions, or prefer a network-level intercept to touch zero handlers?
  • receiveFillfasterWebhook — port its raw entity-API calls to asServiceRole now (so the PDF-download variant works on staging), or stay on the already-portable receiveSignedDocument only?
  • Dev vs staging parity — same mock gateway for both, or a local stub for pure-dev?
