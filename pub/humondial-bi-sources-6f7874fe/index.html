Humondial — מקורות הרשמה וקליקים מ-Short.io

מדריך גישה ל-BI · משלים את מדריך ה-BI הראשי · עודכן 2026-06-14
בקצרה — שתי מערכות נפרדות:
  • קליקים על הלינקים (כמה לחצו על הפלייר / האינסטגרם / וכו') חיים ב-Short.io — לא ב-Supabase. זהו ראש המשפך (חשיפה).
  • הרשמות בפועל (מי נרשם ומאיזה מקור/סניף) חיים ב-Supabase, בעמודה origin_source / origin_branch. זהו תחתית המשפך (המרה).
  • מפתח החיבור בין השתיים הוא short_io_slug שמוגדר לכל מקור בטבלת promo_campaigns.registration_sources.

תוכן עניינים

1. מודל הנתונים — איפה כל דבר יושב 2. מפתח החיבור (short_io_slug) 3. רשימת המקורות + קטגוריות 4. הסניפים 5. גישה ל-Supabase (חיבור + JWT) 6. שאילתות SQL מוכנות 7. דוגמאות cURL (REST) 8. גישה לקליקים ב-Short.io 9. דברים שחשוב לדעת (Gotchas)

הקורא מימין לשמאל: כל לינק שיווקי (Short.io) טוען עם פרמטר ?source=<slug>. כשנגלל עד הרשמה, ה-slug נשמר פעם אחת על השחקן ב-origin_source. מקורות מסוג "סניף" מבקשים גם בחירת origin_branch. הקליקים עצמם — כמה אנשים לחצו אך לא בהכרח נרשמו — קיימים רק ב-Short.io.

2. מפתח החיבור — short_io_slug

בתוך promo_campaigns.registration_sources כל מקור הוא אובייקט עם השדות:

{
  "slug":            "flyer",          // הערך שמגיע ב-?source= ונשמר ב-origin_source
  "label_he":        "פלייר",          // שם לתצוגה
  "short_io_slug":   "FLYER",          // ← מזהה הלינק ב-Short.io (מפתח החיבור)
  "active":          true,
  "requires_branch": false,            // האם דורש בחירת סניף בהרשמה
  "sort_order":      15
}

כדי לחבר קליקים (Short.io) אל הרשמות (Supabase): התאימו את short_io_slug של המקור ללינק התואם ב-Short.io, ואת ספירת ההרשמות שלו לפי origin_source = slug.

6. שאילתות SQL מוכנות

א. הרשמות לפי מקור (כולל label)

SELECT p.origin_source,
       s.label_he,
       s.short_io_slug,
       count(*) AS registrations
FROM   promo_players_bi p
LEFT JOIN LATERAL (
  SELECT (e->>'label_he')      AS label_he,
         (e->>'short_io_slug') AS short_io_slug
  FROM   promo_campaigns c,
         jsonb_array_elements(c.registration_sources) e
  WHERE  e->>'slug' = p.origin_source
  LIMIT 1
) s ON true
GROUP BY 1,2,3
ORDER BY registrations DESC;

ב. הרשמות לפי קטגוריה (המיפוי המומלץ)

SELECT CASE origin_source
         WHEN 'sponsored' THEN 'paid'
         WHEN 'instagram' THEN 'organic_social'
         WHEN 'tiktok'    THEN 'organic_social'
         WHEN 'facebook'  THEN 'organic_social'
         WHEN 'youtube'   THEN 'organic_social'
         WHEN 'whatsapp'  THEN 'organic_social'
         WHEN 'television' THEN 'restaurant'
         WHEN 'shirt'     THEN 'restaurant'
         WHEN 'kappa'     THEN 'restaurant'
         WHEN 'flyer'     THEN 'restaurant'
         WHEN 'website-banner'   THEN 'website'
         WHEN 'website-icon'     THEN 'website'
         WHEN 'website-branches' THEN 'website'
         WHEN 'shmanman-banner'  THEN 'crm_internal'
         WHEN 'workers'   THEN 'crm_internal'
         WHEN 'dailycodes' THEN 'crm_internal'
         ELSE 'direct / untracked'
       END AS category,
       count(*) AS registrations
FROM   promo_players_bi
GROUP BY 1
ORDER BY registrations DESC;

ג. הרשמות לפי סניף (מקורות מסוג סניף)

SELECT b.name AS branch_name,
       count(*) AS registrations
FROM   promo_players_bi p
JOIN   promo_campaigns c ON true
CROSS JOIN LATERAL jsonb_array_elements(c.branches) bx
JOIN   LATERAL (SELECT bx->>'name' AS name, bx->>'tabit_org_id' AS org) b
       ON b.org = p.origin_branch
WHERE  p.origin_branch IS NOT NULL
GROUP BY 1
ORDER BY registrations DESC;

ד. KPI יומי לכל סניף (מוכן, ללא JSON)

SELECT business_date, branch_name,
       new_joins, redemptions, mains_served,
       acquisition, redemption
FROM   promo_branch_daily_stats
ORDER  BY business_date DESC, branch_name;

ה. משפך: קליקים (Short.io) מול הרשמות (Supabase)

הרשמות מ-Supabase לפי short_io_slug; את עמודת הקליקים ממלאים מ-Short.io (סעיף 8) ומחברים על אותו slug:

SELECT s.short_io_slug,
       s.label_he,
       count(p.id) AS registrations   -- clicks ← מ-Short.io, join על short_io_slug
FROM   promo_campaigns c,
       jsonb_array_elements(c.registration_sources) e
       CROSS JOIN LATERAL (SELECT e->>'slug' slug,
                                  e->>'label_he' label_he,
                                  e->>'short_io_slug' short_io_slug) s
LEFT JOIN promo_players_bi p ON p.origin_source = s.slug
WHERE  (e->>'active')::bool
GROUP BY 1,2
ORDER BY registrations DESC;

7. דוגמאות cURL (PostgREST / REST)

בסיס ה-REST: https://qnbogoyoqbkewkdbhxuy.supabase.co/rest/v1. נדרש JWT שנושא {"role":"bi_readonly"} (נטבע מסוד ה-JWT של הפרויקט ע"י אדמין — לא נגיש דרך SQL). שמים אותו גם ב-apikey וגם ב-Authorization.

הרשמות לפי מקור

curl -s "https://qnbogoyoqbkewkdbhxuy.supabase.co/rest/v1/promo_players_bi?select=origin_source" \
  -H "apikey: $BI_JWT" \
  -H "Authorization: Bearer $BI_JWT"
# צד-לקוח: קיבוץ לפי origin_source. ל-aggregation בצד-שרת השתמשו ב-RPC/SQL (סעיף 6).

ה-allowlist של המקורות (slug ↔ short_io_slug ↔ label)

curl -s "https://qnbogoyoqbkewkdbhxuy.supabase.co/rest/v1/promo_campaigns?select=registration_sources&id=eq.bf987a45-9783-4507-9a26-acfd5f145473" \
  -H "apikey: $BI_JWT" \
  -H "Authorization: Bearer $BI_JWT"

KPI יומי לפי סניף

curl -s "https://qnbogoyoqbkewkdbhxuy.supabase.co/rest/v1/promo_branch_daily_stats?select=business_date,branch_name,new_joins,redemptions,mains_served&order=business_date.desc" \
  -H "apikey: $BI_JWT" \
  -H "Authorization: Bearer $BI_JWT"
תקרת REST: PostgREST מחזיר עד 1000 שורות בכל קריאה (ללא קשר ל-Range). לסכומים — העדיפו SQL/RPC על-פני שליפת הכל וקיבוץ ב-JS, או דפדפו עם Range.

8. גישה לקליקים ב-Short.io

הקליקים אינם ב-Supabase. מקור האמת הוא Short.io. שתי דרכים:

א. דאשבורד

נכנסים ל-app.short.io → הדומיין של הקמפיין → Statistics. לכל לינק (לפי ה-short_io_slug מסעיף 3) יש Clicks, Unique clicks, פילוח לפי מקור/מכשיר/זמן. בקשו מהאדמין הזמנה ל-workspace אם אין גישה.

ב. Short.io API

# רשימת הלינקים בדומיין (לאתר את ה-link id לפי ה-path/slug)
curl -s "https://api.short.io/api/links?domain_id=$DOMAIN_ID&limit=150" \
  -H "Authorization: $SHORTIO_API_KEY"

# סטטיסטיקת קליקים ללינק בודד בטווח תאריכים
curl -s "https://api-v2.short.io/statistics/link/$LINK_ID?period=custom&startDate=2026-06-01&endDate=2026-06-30" \
  -H "Authorization: $SHORTIO_API_KEY"

את $SHORTIO_API_KEY מנפיקים ב-Short.io → Settings → Integrations & API. מחברים את הקליקים להרשמות ב-Supabase דרך התאמת path/slug של הלינק ל-short_io_slug.

שים לב להתאמת אותיות: ה-short_io_slug אצלנו נשמר ב-PascalCase (למשל Instagram, FLYER) בעוד origin_source ב-Supabase הוא lowercase (instagram, flyer). בצעו את ההתאמה case-insensitive.

9. דברים שחשוב לדעת

  • קליקים ≠ הרשמות. Short.io סופר לחיצות (כולל בוטים/חזרות); Supabase סופר הרשמות שהושלמו. אל תחשבו יחס המרה בלי לתאם טווחי תאריכים.
  • origin_source/origin_branch נכתבים פעם אחת בהרשמה הראשונה ולעולם לא נדרסים.
  • חלון יום ב-promo_branch_daily_stats: חיתוך 06:00 שעון ישראל (תואם Tabit businessDate).
  • שתי שורות campaign. ב-DB יש שורת קמפיין נוספת (decoy). תמיד סננו לפי id = bf987a45-9783-4507-9a26-acfd5f145473.
  • NULL הוא הקבוצה הגדולה. רוב ההרשמות עדיין ללא מקור מתויג — הציגו אותה כ-"direct/untracked", לא כאפס.
