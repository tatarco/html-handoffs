A single live invocation would time out on file uploads. The function is designed for batched, resumable execution — not one-shot bulk upload.

Controlled migration design — idempotency, visibility, eventual completion

Three phases

PhasePurposeDefault batch size (live)Est. batches for this dataset
analyzeDry run only — full report, zero writesall rows1 invocation
employeesCreate/update Employee records40 employees30 batches (1,170 ÷ 40)
filesDownload Monday assets → BizFlow storage → EmploymentDocument8 employees147 batches (1,175 ÷ 8)

Idempotency guarantees

  • Employee identity key: national ID (idNumber) normalized to digits. Existing BizFlow employees are flagged and skipped unless force_merge: true.
  • Employee upsert on force_merge: updates the existing row by ID — no duplicate employee created.
  • File dedup: before upload, loads existing EmploymentDocument rows for the employee and skips any file whose name already exists. Re-running file phase is safe.
  • Inner join filter: only employees present on both Monday boards are processed — prevents partial/orphan imports.
  • confirm_secret gate: live writes require env secret — prevents accidental execution.

Progress tracking & visibility

  • run_id + AiGenerationJob: each live run stores summary, actionable_total, last_batch_offset, next_batch_offset, and updatedAt on a job row keyed by runId.
  • Per-batch response: every invocation returns summary.batch_processed, batch_has_more, total_files_uploaded_this_batch, total_files_skipped_already_exist.
  • auto_continue: when batch_has_more is true, the function fire-and-forgets the next batch with an incremented batch_offset — a self-chaining pipeline.
  • migrateFromMondayWatchdog CRON: every 5–10 min, finds stale pending migration jobs (no update for 10+ min) and resumes from next_batch_offset. This is the safety net if a chain link fails.

Eventual consistency model

  • Each batch is atomic at the employee level: an employee is created/updated, then their files are uploaded in the same batch (employees phase) or in a dedicated files phase.
  • Failed file downloads produce warnings on the employee result — they do not roll back the employee record. You can re-run file phase to fill gaps.
  • Re-running any phase re-reads Monday boards and BizFlow master data fresh — mapping stays current even if Monday data changes mid-migration.
  • Completion criterion: AiGenerationJob.status = done and batch_has_more = false. Watchdog marks jobs done when next_batch_offset is null.
