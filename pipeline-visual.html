4 — The Database Model (Why JSONB)

Every entity table looks like this

CREATE TABLE candidate ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), company_id UUID NOT NULL, -- tenant isolation doc JSONB NOT NULL, -- ALL fields live here created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ ); -- GIN index for doc @> containment queries -- B-tree index on company_id for RLS -- RLS: company_id = (auth.jwt()->'app_metadata'->>'company_id')

168 Base44 entities → 168 tables. Same 4-column structure each time. No schema mapping needed.

What @bizflow/client exposes

const db = buildEntityMap(supabase, ['candidate']); // Same API as Base44 await db.candidate.list() await db.candidate.get(id) await db.candidate.create({ name: 'Avi' }) await db.candidate.update(id, { status: 'hired' }) await db.candidate.filter({ status: 'new' }) // update() uses jsonb_patch RPC: // UPDATE candidate SET doc = doc || $patch // WHERE id = $id RETURNING id, doc // Merge-patch: only changed fields, nothing destroyed

Gal's app code doesn't need to change — same API shape as Base44's entity client.

Multi-tenancy is enforced in the DB, not the app. RLS policies check auth.jwt()->'app_metadata'->>'company_id' on every read/write. A user can only see their own company's data. app_metadata is server-set (via admin API only) — a user cannot forge their own company_id.
5 — CI Pipeline (supabase-staging.yml — triggers on main push)
1
Gal edits an entity in Base44 UI
Base44 commits the change to base44/entities/EntityName.jsonc on main. GitHub Actions triggers.
2
Codegen: entity JSONC → SQL migration
node scripts/codegen-migrations.mjs — reads all 168 JSONC files, runs the lifecycle state machine (NEW / UNCHANGED / RENAMED / DELETED / RESTORED), emits only the migrations that are needed. Idempotent: re-running on unchanged entities produces nothing.
done: 0 created, 168 skipped ← stable run (nothing new) done: 1 created, 167 skipped ← after adding a new entity
3
Push migrations to Supabase staging
supabase db push --db-url $STAGING_SUPABASE_DB_URL --include-all — applies any new migration files. Supabase CLI tracks which migrations have already run.
4
RLS guard: verify all tables are protected
SQL query: any table in public with no RLS policies → build fails. Prevents shipping tables that are open to all authenticated users.
5
E2E tests
node tests/e2e/run.mjs — three suites: smoke (5 tables + jsonb_patch), CRUD (full lifecycle on candidate), RLS (anon key sees 0 rows). Cleanup always runs even on failure.
6
Open staging → production PR
If all steps pass, GitHub Actions opens (or updates) a PR from main to production. Merging that PR triggers the production deploy. Manual review required.

7 — Admin Panel + User Impersonation

Auth layers

①
Super admin (hardcoded)
Created once with node scripts/setup-super-admin.mjs --email admin@bizflow.app --password ...
Has app_metadata: { role: 'superadmin' } — checked by Edge Function on every admin API call.
②
Super admin creates a company + first user
Admin Dashboard (/admin) → "Create company" → company name + first user email + password → Edge Function creates company row + Supabase Auth user with app_metadata.company_id.
③
Company admin invites more users
/admin/companies/:id → "Invite user" → email → Edge Function calls supabase.auth.admin.inviteUserByEmail + stores invite in company_invites table with the correct company_id.

Impersonation flow

①
Super admin sees floating "A" chip (bottom-right)
Always visible when logged in as superadmin. Click → popover opens with list of all companies + users.
②
Click "Impersonate" next to any user
Auth context saves current admin JWT to sessionStorage.__bizflow_admin_session. Calls Edge Function POST /bizflow-admin/impersonate with target user_id.
③
Magic link redirect
Edge Function calls supabase.auth.admin.generateLink({ type: 'magiclink', email }) → returns a one-time URL → frontend navigates there → Supabase sets the new user's session automatically.
④
Orange banner appears above EVERYTHING
ImpersonationBanner is mounted at the root of the app (outside all routes). Detects sessionStorage.__bizflow_admin_session → shows "🎭 Acting as: user@company.com — [Exit]".
Exit → supabase.auth.setSession(savedAdminSession) → banner goes away.

Edge Function: bizflow-admin

Lives at supabase/functions/bizflow-admin/index.ts. All routes validate app_metadata.role === 'superadmin' from the JWT before doing anything. Uses service role key server-side (never exposed to the browser).

GET /companies POST /companies GET /companies/:id/users POST /companies/:id/invite POST /impersonate
